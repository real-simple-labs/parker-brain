# Review queue — proposals pending Jimmy's approval

This folder is the single place Jimmy looks to see everything a knowledge source would change, before it changes. Its job is to make sure Jimmy never has to dig, and never has to imagine the change.

Knowledge intake is **approval-gated and propose-first**. When Parker learns something from a source, it does **not** modify the real surface. It writes the **exact, verbatim change it would make** — the full new-doc content, or the precise old → new for an edit — as a proposal entry here, and surfaces the same plan to Jimmy. Nothing touches the live context docs, knowledge docs, skills, or prompts until Jimmy approves. The only thing written without approval is the provenance receipt in `inbox/`, which records the source, not a change to Parker's knowledge.

On Jimmy's call: **approve** → Parker applies the proposed text exactly and marks the entry `applied`; **modify** → Parker applies Jimmy's adjusted version; **deny** → nothing is written to the surface and the entry is marked `denied` with the reason.

## The states

- **Proposed — pending Jimmy.** Parker has written the exact change here but has not touched the real surface. Jimmy approves, modifies, or denies.
- **Watch.** The source is too thin (single, mixed-confidence) to justify a change yet. The entry names the change it *would* propose and the promotion condition that would earn it.
- **No change warranted.** The source duplicates what Parker already holds; the entry says so and points to where it already lives. Anti-bloat is a feature.
- **Applied / denied.** The resolved state after Jimmy's decision.

## What every queue entry names

- the source signal it came from
- which of the four components it touches: context doc, knowledge doc, tool call, or process
- the exact target surface(s) and, for a proposed change, the **verbatim** edit (old → new, or full new-doc content)
- confidence and, for watch items, the promotion condition
- state: proposed / watch / no-change / applied / denied
