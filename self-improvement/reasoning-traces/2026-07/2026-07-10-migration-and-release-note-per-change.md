# 2026-07-10 — Every change ships its versioning artifacts, not just the change

**Type:** process rule / standing self-note. **Source:** Anton, 2026-07-10, after a scriptwriting-gate PR was opened *without* a migration or release note and had to be corrected. Explicit instruction: "store in your memory that you have to do this (migrations / versioning) each time for each change."

**The rule.** Do not treat a change as done at the edit. Before opening or updating a PR, ask the versioning questions and ship the artifacts the answer demands:

- **Every release ships a `migrations/vN.md`, named for its tag — but most are one-liner no-ops.** The question is only whether the migration has *real steps* or is a no-op record. Real steps are needed when an already-built brain must *do* something the sync script can't: a renamed/moved path a brain references, a new standing file **outside** the copied bundle, a prompt re-run, or an edit to brand-authored content. A change that lives **entirely inside the copied bundle** — skills, agents, hooks, checker scripts, schedule recipes (the `scripts/sync-executable-layer.py` BUNDLE_MAP) — writes the no-op: "Nothing to do; the pin bump's re-sync delivers it. Record `vN`." A migration must **never** copy or diff a path the sync script owns; check the BUNDLE_MAP first and delete any such step. A pure method change loaded straight from the mount (`prompts/`, `creative-strategy-context/`, a system doc) is also a no-op. (Canonical: `migrations/README.md`, CLAUDE.md "Releases And Migrations," `update-parker-skill` checklist item.)
- **Add a `release-notes/` entry** for any factory change that reshapes standing brains — dated `YYYY-MM-DD-<slug>.md`. If the target release isn't cut yet, record the change against the pending `vN` note (create it if absent, append if it exists) and mark it in-progress; the release may accumulate more before the tag.
- **State the data-transformation posture explicitly** in the migration. Text/method-only changes say "no data transformation needed; no brand data touched." Only real brand-data conversions get transform steps, and those are offered before applying.
- **Tags are manual.** Never cut `git tag vN` unless asked; the migration and release note wait in the PR until the human cuts the tag.
- **Keep both artifacts tight, and don't duplicate mechanism.** A migration names only what is brain-specific for *this* change — it does not re-describe how to check out a tag, re-sync the executable layer, or verify; `/update-brain` and `scripts/sync-executable-layer.py` own that. A release note is *what shipped* and *why*, plus the migration pointer. No follow-up lists, no enforcement-roadmap riffs, no lengthy design discussion — those belong in reasoning traces or issues, not in release notes. (Anton, 2026-07-10: "do not put lengthy discussions, followups etc into release notes.")

**Why I missed it the first time.** I scoped the task to the skill wording and stopped at the prompt edit, treating a `.claude/skills/` change as method-only. But skills live in the committed `.claude/` bundle and are copied into each brain, so they are exactly the "changes the committed `.claude/` bundle" case that requires a migration. The lesson generalizes: the versioning artifacts are part of the change, not a follow-up.
