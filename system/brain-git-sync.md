# Brain git sync — how a brand brain talks to GitHub, and how we make agents do it right

The runtime procedure lives in the routine bundle as the `save-brain` skill (`templates/brand-routines/claude/skills/save-brain/SKILL.md`, shipped to brains at `.claude/skills/save-brain/`). This doc is the maintainer's side: the facts the procedure rests on, the enforcement stack that makes agents actually follow it, and why each piece exists. Change the procedure there, check the reasoning here.

## The facts underneath (from the Parker MCP server)

- Managed brains live in the **`parker-brain` GitHub org** (server env `GITHUB_PARKER_ORG`), one private repo per brand, named `<org>-<brand>` or `<brand>`, provisioned by `setup_parker_brain`.
- Every `setup_parker_brain` call re-mints a **GitHub App installation token**: ≤1 hour, scoped to that single repo, `contents: write` only. It arrives embedded in `authenticated_clone_url` as `https://x-access-token:<TOKEN>@github.com/parker-brain/<repo>.git` — the agent lifts the token out; that URL is never fed to git. There is no rotation — expired means call the tool again.
- Human collaborators are added at write permission via the same tool (`github_username` param, invitation flow). Their personal auth is for *their* clones on *their* machines — never for the agent.
- Repos have **no branch protection** and the default branch is whatever the first push created (`main`). This is why the procedure bans force-pushes and branches outright rather than relying on GitHub to stop them.

## Why the agent must never use the user's auth or `gh`

The user usually has no working GitHub auth in the session at all (they were invited, or never touched GitHub). When they *do*, using it misattributes commits, sidesteps the access Parker manages, and — for `gh` specifically — habituates a tool that will be missing or wrongly-logged-in in most brand sessions. One credential path means one failure mode and one fix: re-call the tool.

## Credentials live in `.git/parker-credentials` (and how we got here)

`origin` carries the **plain** URL (`https://github.com/parker-brain/<repo>.git`), and the token lives in git's own store-helper file at `.git/parker-credentials` — one line, `https://x-access-token:<TOKEN>@github.com` — wired by a one-time, two-entry repo setting: `git config credential.helper ""` then `git config --add credential.helper "store --file .git/parker-credentials"`. The blank first entry is load-bearing, not decoration: helper lists accumulate across config scopes, and macOS git ships a system-level `osxkeychain` helper that would otherwise answer *first* with the user's personal GitHub credential — pushing to the managed repo as them, the exact misattribution this design bans (v6 never hit this because URL-embedded credentials outrank all helpers). An empty entry clears the inherited list, so only the store file remains. The agent writes that file with the **Write tool**, never through a shell command. Plain `git pull --rebase origin main` and `git push origin main` work for the token's hour (git consults the helper only when GitHub answers 401, so the public mount's fetches never even send it); when one fails with an auth error, the playbook is: call `setup_parker_brain` again, re-Write the file, retry. Explicit `origin main` on every push so nothing depends on upstream config that a fresh clone doesn't have. Cloning uses the same rail before the repo exists: credential line in a temp file, `git -c credential.helper= -c credential.helper="store --file <temp>" clone --recurse-submodules <plain url>` (same blank-entry trick, inline), then the file moves into `.git/` and the repo config gets the permanent wiring.

This is the third revision, and each move was forced. v5 bracketed every sync with token-in/token-out `set-url` calls so no secret lingered in `.git/config`; the ceremony was worse than the risk — agents skipped pushes or stalled asking the user — so v6 left the tokenized URL sitting in `origin`, accepting a secret that is local-only, single-repo, `contents:write`, and dead within the hour. v6 then died on a different wall: **Claude Code's permission classifier refuses any Bash command that carries a live token** ("Credential Leakage", blocked regardless of destination trust or brevity of exposure), and in the token-in-remote design the refresh itself — `git remote set-url origin https://x-access-token:ghs_…` — is exactly such a command. So was the tokenized clone. The credential file keeps everything v6 bought (no ceremony around pushes, same risk envelope: the file sits inside never-pushed `.git/`) while the token never transits a shell command at all — it moves only through the Write tool, which is git's intended credential-store pattern rather than a workaround. A rejected token even self-cleans: git tells the store helper to erase the stale line on auth failure, and the refresh is one Write instead of one `set-url`.

Two Write-tool facts the procedure accounts for. The Write tool refuses to overwrite a file it hasn't read in the session, so every credential refresh runs `rm -f .git/parker-credentials` (no secret in that command) before the re-Write. And the permission classifier can block a token write it deems un-requested, so the bundle's `.claude/settings.json` pre-approves writes to `.git/parker-credentials` and the clone-time temp file — allow rules are checked before the classifier, which is also what lets headless scheduled runs refresh credentials with nobody watching. If a block still fires (team-edited settings, model-side refusal), the skill's fallback is a plain-words user question when someone is present, and commit-locally-and-report-loudly when nobody is.

## Detecting the self-hosted exception

A rare team hosts the brain in their own GitHub instead of the managed org. Detection is the origin URL: `github.com/parker-brain/…` → managed, anything else → theirs, their auth, `gh` allowed. A repo with no remote yet counts as managed when `parker_config.json` exists (mid-build). Both the skill and the guard hook use exactly this test, so the exception needs no configuration.

## The enforcement stack (instructions alone don't survive contact)

1. **`git-guard.py`** — PreToolUse hook on Bash in the shipped `.claude/settings.json`. Deterministic: on managed repos it blocks `gh`, any command carrying the raw token (`x-access-token:` in the command — the platform classifier would kill it anyway; the guard teaches the file flow first), credential-less `push`/`pull`/`fetch` (satisfied by a non-empty `.git/parker-credentials`, or by a token still in `origin` on a not-yet-migrated brain), any force-push, and clones missing `--recurse-submodules` or the `-c credential.helper` flag; its block messages teach the credential-file refresh at the moment of the mistake. Ops on the public mount (`git -C parker-system …`) pass through. It fails open internally, and its settings entry must never get the `2>/dev/null || true` wrapper the other hooks use — exit code 2 and stderr are the mechanism.
2. **`session-start.py`** — the pull-first enforcement. On session start it *runs* `git pull --rebase origin main` + submodule update itself when the working tree is clean, instead of only reminding; on a dirty tree or a failure (expired token, offline) it reports loudly and makes fixing the pull the session's first job. It deliberately does **not** synchronize factory tags: tags do not affect the checked-out submodule pin, and release discovery belongs only to `/update-brain`, which runs `git -C parker-system fetch origin --prune --prune-tags` before comparing releases.
3. **The `save-brain` skill** — the full procedure, discoverable by name and description for anything save/sync/clone-shaped, and the reference the guard's messages point to. Self-contained on purpose: git trouble often happens exactly when the mount is empty.
4. **The brand `CLAUDE.md` section** ("How this brain saves itself") — always loaded, carries the two absolutes: pull first, and commit-and-push immediately without ever asking the user for permission to save. The push-always discipline is deliberately prompt-level, not a hook: a Stop-hook version was built and removed, because a blocking guard risks trapping a turn behind a persistent git error and can't see an explicit "don't commit" from the user — the two absolutes plus the end-of-turn self-check (and the user-override carve-out) live here and in the skill instead.
5. **The tool's own returned message** (mevin2 side) — the one channel that reaches sessions outside the repo, e.g. claude.ai during onboarding. It should teach the same credential-file flow — and ideally return the token as its own field so no agent has to parse it out of the URL; as of v8 it still teaches clone-the-authenticated-URL / `set-url`-on-expiry, which walks agents straight into the platform's token-in-command block. Keeping it consistent with the skill is a cross-repo duty, and until it catches up, the skill and hooks here override what the tool message says.

Layers 1–4 travel in the routine bundle, so `sync-executable-layer.py` delivers them to standing brains on a pin bump; only a team-customized `settings.json` needs the migration's manual merge.
