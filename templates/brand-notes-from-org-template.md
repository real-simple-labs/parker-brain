# Brand-notes-from-org template

> Status: `[~]` drafted 2026-06-15, pending Jimmy's review. Canonical home per brand: `running-notes/brand-notes-from-org.md` (standalone brains) / `z-brands/[brand]/running-notes/brand-notes-from-org.md`. One of the four always-loaded core memory docs. This is the team's collective state on this brand, synthesized across every user — the org-level memory Parker reads to know how this brand actually works with it.

## What this surface is

The brand's organizational memory: the team-wide truths that hold across users. Where `brand-profile.md` is the research read of the brand and is frozen against chat, refreshed only on a research cadence, this doc is the opposite. It is built from usage. It accumulates from conversations across the whole team — what they bring Parker, how they work with it, what they have decided is true, and what they are driving toward — and it is always loaded, so every chat starts from the team's shared state rather than a blank slate.

It is maintained, not regenerated. The memory extraction pass reads recent conversations across all users on the brand and updates the section it touches; nothing here comes from the research build. It is org-scoped: a fact true for one user but not the team belongs in that user's `brand-notes-from-user.md`, not here.

## Structure

```
---
brand:
doc: brand-notes-from-org
last_updated:
sources: memory extraction across all users' recent conversations on this brand
---

# Brand notes from org — [brand]

## What the brand uses Parker for
## How the team uses Parker
## Current work
## What success looks like
## Brand rules
## Recent validations
```

**What the brand uses Parker for** — the jobs the team brings Parker: the recurring asks, the workflows it sits inside, the roles that lean on it. The use-case read, not the usage mechanics.

**How the team uses Parker** — usage user by user: who uses it for what, where it is working, where it is underused, and the patterns worth spreading. This is where gaps and inspiration across the team surface.

**Current work** — what the team is collectively in flight on right now, read across recent chats. The present tense of the account. Forward-looking, per-user awareness of upcoming work lives in `brand-notes-from-user.md`, not here.

**What success looks like** — the running definition of success for this brand, drawn from chats and the org's stated goals. The north star later work is measured against.

**Brand rules** — the explicit DOs and DON'Ts: fact-based gates, legal and compliance constraints, hard guardrails. Distinct from user preferences. A brand rule binds everyone, a banned claim or a required substitution; a preference is one person's taste and lives in their user docs. State each as a gate with its reason, not a suggestion.

**Recent validations** — the narrative of what just got validated through the open-loops pipeline and where that says the brand should keep focus. Points to the canonical validations; does not restate them.

## Rules

1. Org-scoped only. A truth that holds for the team belongs here; a single user's pattern, preference, or learning belongs in that user's `brand-notes-from-user.md`.
2. From usage, never from research. This doc never absorbs what `brand-profile.md` already carries. If a fact is a research finding about the brand or market, it lives in the profile.
3. Maintained in the memory pass: each extraction updates the touched section with a date; stale entries are retired, not left to rot.
4. Brand rules outrank preferences and are stated as gates. A rule with a reason and a date beats a vague guardrail.
5. Quote and point. Validations point to the canonical validation docs; current work points to the live threads. This is the synthesis layer, not the archive.
