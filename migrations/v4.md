# v4 — the fast update: script-driven re-sync of the copied layer

**Applies to:** brains pinned at `v3` or earlier (after prior migrations). No-op on `v4+`.

**What changes:** the manual per-file re-sync of the copied executable layer is replaced by a deterministic script, `parker-system/scripts/sync-executable-layer.py`. It applies the three piles mechanically — untouched copies refresh silently, team-edited files stay theirs and are listed, nothing is deleted — so a routine update becomes one question and one short summary.

## Steps

1. **Add the "where brand adaptations live" note to the brand's `CLAUDE.md`** — this edits brand-authored content, so show it to the team before applying, per the offer rules. If `CLAUDE.md` already says adaptations belong in brand docs first, skip. Otherwise, in the section that tells the agent to refine or adapt the skills to the brand, append:

   > When a skill needs adapting to this brand, put the adaptation in the brand's own docs first — `brand-lens.md`, the brand rules, the script-voice profile, running notes — because the skills read those at runtime and factory updates never touch them. Edit a copied skill or schedule file directly only when the adaptation truly can't live in a brand doc; a file you edit is yours from then on, and updates will leave it frozen at your version (you'll see it listed as "left alone" in each update summary).

2. **Record it:** note migrations applied through `v4` in `running-notes/standard-sync.md` and set `parker_brain_version` to `v4` in `parker_config.json`.

## How to perform this bump's re-sync (a note to the updating agent, not a migration step)

`/update-brain` runs one re-sync of the copied executable layer after all migrations finish. From this release on, that re-sync **is** the script — when you reach it, run:

```bash
python3 parker-system/scripts/sync-executable-layer.py --from <old pin>
```

Do not compare the copied files by hand, and do not run the script inside this migration — it runs once, after the migrations, per `migrations/README.md`. It refreshes every unmodified copy (including `/update-brain` itself, which is how the new behavior arrives) and re-running it is a safe no-op.
