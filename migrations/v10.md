# v10 — credential-write hardening: pre-approval rules and the rm-first refresh

**Applies to:** brains pinned at `v9` or earlier.

**What changes:** the credential file flow from `v8` gains two field fixes — a `permissions.allow` block in `.claude/settings.json` that pre-approves writes to `.git/parker-credentials` (so the safety classifier is never asked about the one sanctioned token location), and an `rm -f .git/parker-credentials` step before every credential re-Write (the Write tool refuses to overwrite a file it hasn't read). Both travel inside the copied bundle, so the pin bump's normal re-sync delivers them.

**One check on the brand side:** if the re-sync lists `.claude/settings.json` as team-edited (kept, not refreshed), merge this block into its `permissions` by hand, keeping the team's customizations:

```json
"allow": [
  "Write(.git/parker-credentials)",
  "Write(/.git/parker-credentials)",
  "Write(//tmp/parker-credentials)",
  "Bash(rm -f .git/parker-credentials)"
]
```

Otherwise nothing to do. Record migrations through `v10` and set `parker_brain_version` to `v10` as usual.
