# v8 — credentials move out of the remote URL into `.git/parker-credentials`

**Applies to:** brains pinned at `v7` or earlier. The conversion steps are for managed brains (origin under `github.com/parker-brain/`); a self-hosted brain just records the version.

**What changes:** the short-lived token no longer lives inside the `origin` URL, because Claude's own safety layer now blocks any shell command that carries a live token — which killed the old refresh (`git remote set-url origin <tokenized url>`) and tokenized clones outright. Credentials move to git's store file at `.git/parker-credentials`, written with the **Write tool**, and `origin` goes back to the plain URL. The updated `save-brain` skill and hooks arrive with the pin bump's normal re-sync; the steps below are the one-time local conversion the sync script can't do, because it is repo state rather than a bundled file. Full procedure afterwards: `/save-brain`.

## Steps (managed brains only)

1. Point `origin` at the plain URL — this command carries no secret and is safe to run:
   `git remote set-url origin https://github.com/parker-brain/<repo>.git`
   (Take `<repo>` from the current origin URL. If the origin already has no `x-access-token:` in it, skip this step.)
2. Wire the credential file, one time — two commands, in this order:
   `git config credential.helper ""`
   `git config --add credential.helper "store --file .git/parker-credentials"`
   (The blank first entry shuts out any system or global helper — like the user's macOS keychain — that would otherwise answer first with *their* personal GitHub credential and misattribute pushes.)
3. Mint the credentials into the file: call `setup_parker_brain` (Parker MCP), lift the token (`ghs_…`) out of the returned `authenticated_clone_url`, and use the **Write tool** to save this single line to `.git/parker-credentials`:
   `https://x-access-token:<TOKEN>@github.com`
   Never put the token itself into any shell command — not in a URL, not in an echo.
4. **Do not run any pull, fetch, or push between step 1 and the end of the update.** The pre-v8 `git-guard` hook is still live while migrations run (the hook re-sync happens after), and it blocks network git commands that don't carry the old token-in-origin. Verification happens naturally at the update's close: once the re-sync delivers the v8 hooks, the routine's normal closing save (`git pull --rebase origin main`, `git push origin main` per `/save-brain`) exercises the new credentials. If that hits an auth error, re-do step 3 with a fresh token and retry.

Record migrations through `v8` and set `parker_brain_version` to `v8` as usual.
